# Guide SQL Avancé — Maîtriser les Requêtes Imbriquées, les CTE et les Vuess

## Introduction

Vous maîtrisez désormais la création de bases de données, l'interrogation via des requêtes simples ainsi que l'usage des jointures relationnelles. Cependant, à mesure que les besoins d'analyse se complexifient, vos requêtes prennent de l'ampleur. Les sous-requêtes, les expressions de table communes (CTE) et les vues sont des outils indispensables pour segmenter votre logique, optimiser les calculs et maintenir un code SQL lisible et évolutif.

> 💡 **Analogie** : Si les requêtes SQL de base s'apparentent à des phrases simples, les sous-requêtes, les CTE et les vues fonctionnent comme des paragraphes et des chapitres structurés. Ils organisent votre flux de données pour rendre votre logique métier limpide.

---

## Objectifs d'apprentissage

* 🎯 **Maîtriser les sous-requêtes** pour imbriquer des logiques de filtrage et de calcul dynamique.
* 🎯 **Implémenter la clause `WITH` (CTE)** afin d'isoler des blocs de calcul temporaires et structurer le code de manière séquentielle.
* 🎯 **Créer et manipuler des Vues (`VIEW`)** pour mettre à disposition des tables virtuelles réutilisables.
* 🎯 **Garantir la maintenabilité** et la lisibilité des scripts d'extraction selon les standards de l'ingénierie de données.

---

## Problématique : La complexité du SQL

En enchaînant les opérations de sélection, d'agrégation, de filtrage et de jointures multiples, un script SQL peut rapidement devenir un bloc monolithique de plusieurs dizaines de lignes.

> ⚠️ **Point de vigilance** : Un script SQL trop long et dépourvu de structure logique devient une boîte noire impossible à déboguer, à optimiser ou à maintenir, y compris pour son propre auteur après quelques semaines.

Pour pallier ce problème, le langage SQL offre des mécanismes permettant de découper le code en unités logiques distinctes.

## 1. Les Sous-requêtes : Des requêtes imbriquées

### Définition et exécution

Une sous-requête est une instruction `SELECT` encapsulée à l'intérieur d'une autre requête SQL (principale ou parente). Le SGBD résout d'abord la sous-requête afin d'utiliser son résultat comme paramètre d'entrée pour la requête globale.

```
+--------------------------------------------------------+
| REQUÊTE PRINCIPALE (Parente)                           |
|                                                        |
|   SELECT ... FROM ... WHERE colonne IN (               |
|       +--------------------------------------------+   |
|       | SOUS-REQUÊTE (Imbriquée)                   |   |
|       |                                            |   |
|       | SELECT colonne FROM table WHERE ...        |   |
|       +--------------------------------------------+   |
|   );                                                   |
+--------------------------------------------------------+

```

### Caractéristiques fondamentales

* Elles sont obligatoirement délimitées par des parenthèses `( )`.
* Elles s'exécutent en priorité avant la requête qui les contient.
* Elles s'intègrent principalement dans les clauses `WHERE`, `HAVING`, `FROM` et `SELECT`.

### Exemple d'application

Extraction des clients ayant réalisé au moins un achat d'un montant unitaire supérieur à 500€ au sein d'une base relationnelle :

```sql
SELECT 
    firstname, 
    lastname
FROM 
    customer
WHERE 
    id IN (
        SELECT customer_id
        FROM sale
        WHERE id IN (
            SELECT sale_id
            FROM sale_product
            WHERE product_id IN (
                SELECT id
                FROM product
                WHERE price > 500
            )
        )
    );

```

### Typologies de sous-requêtes

#### A. Sous-requête scalaire

Retourne une valeur unique (une seule ligne et une seule colonne). Elle s'associe aux opérateurs de comparaison classiques (`=`, `>`, `<`, `!=`).

*Exemple : Identifier le client ayant effectué la toute première commande enregistrée :*

```sql
SELECT 
    firstname, 
    lastname
FROM 
    customer
WHERE 
    id = (
        SELECT customer_id
        FROM sale
        ORDER BY sale_date ASC
        LIMIT 1
    );

```

#### B. Sous-requête colonne

Retourne une liste de valeurs issues d'une unique colonne. Elle nécessite l'usage d'opérateurs d'ensembles comme `IN`, `NOT IN`, `ANY`, `ALL` ou `EXISTS`.

*Exemple : Identifier les clients ayant acheté des articles de la catégorie "Composants PC" :*

```sql
SELECT 
    firstname, 
    lastname
FROM 
    customer
WHERE 
    id IN (
        SELECT customer_id
        FROM sale
        WHERE id IN (
            SELECT sale_id
            FROM sale_product
            WHERE product_id IN (
                SELECT id
                FROM product
                WHERE category_id = (
                    SELECT id
                    FROM category
                    WHERE category_name = 'Composants PC'
                )
            )
        )
    );

```

#### C. Sous-requête table (Requête dérivée)

Retourne un ensemble multidimensionnel (plusieurs lignes et plusieurs colonnes). Elle s'injecte directement dans la clause `FROM` et agit comme une table temporaire. **Elle nécessite obligatoirement la définition d'un alias.**

*Exemple : Extraire le chiffre d'affaires cumulé par client :*

```sql
SELECT 
    c.firstname, 
    c.lastname, 
    t.total_spent
FROM 
    customer c
JOIN (
    SELECT 
        s.customer_id, 
        SUM(p.price * sp.quantity) AS total_spent
    FROM 
        sale s
    JOIN sale_product sp ON s.id = sp.sale_id
    JOIN product p ON sp.product_id = p.id
    GROUP BY s.customer_id
) t ON c.id = t.customer_id
ORDER BY t.total_spent DESC;

```

## 2. La clause `WITH` : Common Table Expressions (CTE)

La clause `WITH` permet de nommer et d'isoler des sous-requêtes avant l'instruction principale. Contrairement aux sous-requêtes imbriquées à l'infini qui nuisent à la lisibilité, les CTE permettent de séquencer le code de haut en bas, de manière logique et linéaire.

```sql
-- Définition de la CTE (Table temporaire nommée)
WITH high_value_products AS (
    SELECT id, product_name, price
    FROM product
    WHERE price > 500
)
-- Exploitation immédiate dans la requête principale
SELECT 
    product_name, 
    price
FROM 
    high_value_products
ORDER BY price DESC;

```

### Avantages des CTE par rapport aux sous-requêtes imbriquées

* **Lisibilité** : Évite l'effet "poupée russe" des parenthèses imbriquées.
* **Factoring** : Une même CTE peut être appelée plusieurs fois par des jointures au sein de la requête principale.
* **Maintenance** : Isole la logique d'extraction de la donnée de sa logique de présentation.

## 3. Les Vues (`VIEW`) : Tables virtuelles persistantes

Une vue est une structure logique stockée dans le dictionnaire de la base de données. Elle enregistre une requête `SELECT` complexe sous la forme d'une table virtuelle. Elle ne duplique pas les données physiques : à chaque appel de la vue, la requête sous-jacente est réexécutée sur les tables sources.

### Syntaxe générale

```sql
CREATE VIEW IF NOT EXISTS v_sales_by_category AS (
    SELECT 
        c.category_name,
        SUM(p.price * sp.quantity) AS total_sales,
        COUNT(DISTINCT s.id) AS number_of_orders
    FROM category c
    JOIN product p ON c.id = p.category_id
    JOIN sale_product sp ON p.id = sp.product_id
    JOIN sale s ON sp.sale_id = s.id
    GROUP BY c.id, c.category_name
);

```

### Exploitation d'une vue

Une fois créée, la vue se manipule exactement comme une table physique standard :

```sql
SELECT * FROM v_sales_by_category
WHERE number_of_orders > 10
ORDER BY total_sales DESC;

```

### Synthèse comparative : CTE vs Vues vs Sous-requêtes

| Fonctionnalité | Définition / Périmètre | Persistance en Base | Réutilisabilité |
| --- | --- | --- | --- |
| **Sous-requête** | Locale, confinée à l'intérieur d'une clause spécifique. | Non (volatile). | Uniquement au sein de la ligne de code parente. |
| **CTE (`WITH`)** | Déclarée en amont d'un bloc de script SQL global. | Non (détruite après exécution). | Réutilisable plusieurs fois dans la même requête. |
| **Vue (`VIEW`)** | Objet permanent rattaché au schéma de la base de données. | Oui (la structure est sauvée, pas la donnée). | Appelable par n'importe quel utilisateur ou script tiers. |

## 4. Atelier Pratique : Audit du Catalogue Cinématographique

Cet atelier s'exécute sur l'environnement client **DBeaver** connecté au fichier de base de données SQLite `movies.sqlite`.

### Spécifications techniques des requêtes à implémenter

#### Exercice 1 : Filtrage par valeur scalaire moyenne

Écrire une requête extrayant les titres (`original_title`) et les dates de sortie (`release_date`) des films disposant d'un score moyen d'évaluation strictement supérieur à la moyenne calculée sur l'intégralité du catalogue.

#### Exercice 2 : Filtrage d'ensembles par agrégation

Lister l'ensemble des films produits et réalisés par des cinéastes ayant à leur actif un volume de production strictement supérieur à 3 films enregistrés en base.

#### Exercice 3 : Analyse temporelle croisée via CTE

Isoler les films (titres, dates de production et nom du réalisateur) dont l'année de sortie correspond exactement à celle du film culte *'The Dark Knight'*. L'extraction de l'année s'effectuera à l'aide de la fonction native `strftime('%Y', release_date)`.

* *Contrainte additionnelle : Traiter cette problématique d'abord via une sous-requête classique, puis refactoriser le code à l'aide d'une structure propre en clause `WITH`.*

#### Exercice 4 : Encapsulation et persistance par Vue

1. Initialiser une vue nommée `statistics_directors` compilant les données par réalisateur : nom du cinéaste, volume total de films réalisés et note moyenne pondérée de ses productions.
2. Interroger cette vue pour extraire le Top 3 des réalisateurs les mieux notés de l'histoire, en appliquant un filtre de représentativité égal ou supérieur à 3 films réalisés.
