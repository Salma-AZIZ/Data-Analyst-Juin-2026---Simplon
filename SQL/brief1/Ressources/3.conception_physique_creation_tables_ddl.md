### 1. Le langage SQL

SQL (Structured Query Language) est un langage informatique normalisé permettant d'interagir avec des bases de données relationnelles. Imagine SQL comme une langue commune que tous les systèmes de gestion de bases de données peuvent comprendre.

> [!NOTE]
> **Analogie simple** : Si les bases de données sont des bibliothèques, SQL est le langage que tu utilises pour demander au bibliothécaire de trouver, ranger, ajouter ou retirer des livres.

Créé dans les années 1970, SQL est aujourd'hui normalisé par l'ISO (International Organization for Standardization) afin d'assurer la compatibilité entre différents SGBD. Il permet également de réaliser des opérations complexes comme les jointures multiples, la gestion des transactions et la création de vues pour simplifier les requêtes.

#### Définition et caractéristiques

SQL présente plusieurs particularités :
* **Langage déclaratif** : Tu indiques ce que tu veux obtenir sans préciser comment le SGBD doit réaliser l'opération sous le capot.
* **Optimisation automatique** : Le SGBD optimise l'exécution des requêtes pour fournir le résultat de la manière la plus efficace possible.
* **Standardisation** : SQL est universel et fonctionne avec la majorité des bases de données relationnelles.
* **Extensibilité** : Grâce aux procédures stockées, triggers et vues, SQL permet d'automatiser des tâches répétitives.

> [!WARNING]
> Contrairement à des langages comme Python, SQL ne nécessite pas de détailler l'ordre d'exécution algorithmique des instructions. L'optimiseur de requêtes du SGBD détermine l'ordre le plus efficace pour exécuter tes commandes.

#### Les 4 sous-ensembles du langage SQL

Découvrons les principales familles de commandes SQL que tu utiliseras tout au long de ta carrière :

1. **DQL (Data Query Language) — Requêtes d’interrogation** : Permettent d’extraire des données sans altérer la base (Mot-clé principal : `SELECT`).
2. **DML (Data Manipulation Language) — Requêtes de manipulation** : Permettent d’ajouter, modifier ou supprimer des lignes de données (`INSERT`, `UPDATE`, `DELETE`).
3. **DDL (Data Definition Language) — Requêtes de définition** : Permettent de créer, modifier et détruire la structure même des bases de données et des tables (`CREATE`, `ALTER`, `DROP`).
4. **DCL (Data Control Language) — Requêtes de contrôle** : Permettent de gérer les permissions, les rôles et la sécurité des données (`GRANT`, `REVOKE`).

### 2. Les types de données en SQL

Les types de données structurent et contrôlent les informations stockées dans une base SQL, garantissant intégrité et performance. Choisir le bon type est crucial pour optimiser l’espace de stockage et assurer la fiabilité de tes analyses.

| Catégorie | Types standards courants | Cas d'utilisation |
| :--- | :--- | :--- |
| **Numérique Entier** | `TINYINT`, `SMALLINT`, `INT`, `BIGINT` | Identifiants, quantités, âges, stocks. |
| **Numérique Décimal** | `DECIMAL(taille, dec)`, `FLOAT`, `DOUBLE` | Prix unitaires, taux, mesures physiques exactes. |
| **Chaîne de caractères** | `CHAR(N)`, `VARCHAR(N)`, `TEXT` | Prénoms, emails, codes postaux, descriptions longues. |
| **Date et Heure** | `DATE`, `TIME`, `DATETIME`, `TIMESTAMP` | Dates de transactions, logs de connexions, anniversaires. |

> [!CAUTION]
> Le choix du type de données est essentiel : Un type trop large gaspille de l'espace de stockage et ralentit les index, tandis qu'un type trop restrictif peut entraîner des erreurs de troncature ou la perte de données précieuses.

### 3. Les clés et les contraintes d'intégrité en SQL

#### Qu'est-ce qu'une clé ?
Une clé identifie de manière unique un enregistrement ou relie différentes tables entre elles.

> [!NOTE]
> **Analogie** : Si une base de données est un grand immeuble, les clés sont les numéros d'appartement permettant de retrouver chaque logement avec précision ou d'identifier la boîte aux lettres associée.

#### Les différents types de clés

##### Clé primaire (PRIMARY KEY - PK)
* Identifie de manière unique chaque enregistrement dans une table.
* Ne peut **jamais** contenir de valeurs `NULL`.
* Est souvent associée à un attribut auto-incrémenté.
* Une table possède **une seule et unique** clé primaire.

```sql
CREATE TABLE customer (
    id INT AUTO_INCREMENT PRIMARY KEY,
    firstname VARCHAR(50) NOT NULL,
    lastname VARCHAR(50) NOT NULL
);

```

##### Clé étrangère (FOREIGN KEY - FK)

* Établit une relation d'intégrité référentielle entre deux tables.
* Fait référence à la clé primaire d'une autre table.
* S'assure que les données de liaison référencées existent bel et bien.

```sql
CREATE TABLE sale (
    id INT AUTO_INCREMENT PRIMARY KEY,
    sale_date DATE NOT NULL,
    customer_id INT NOT NULL,
    FOREIGN KEY (customer_id) REFERENCES customer(id)
);

```

> [!TIP]
> **Logique relationnelle** : Dans la relation ci-dessus, la clé étrangère `customer_id` située dans la table `sale` matérialise le fait qu'un client peut effectuer plusieurs ventes, tandis que chaque vente individuelle n'appartient qu'à un seul client.

##### Clé composée (COMPOSITE KEY)

* Formée par la combinaison de deux ou plusieurs colonnes de la table.
* Sert de clé primaire ou d'unicité lorsque l'identité d'une ligne dépend de la réunion de plusieurs critères (très fréquent dans les tables de liaison N:M).

#### Les autres contraintes d'intégrité incontournables

* 📌 `NOT NULL` : Empêche une colonne de contenir des valeurs vides (`NULL`). Idéal pour forcer la présence d'informations indispensables (ex. nom, prix).
* 📌 `UNIQUE` : Garantit que toutes les valeurs d'une colonne sont distinctes les unes des autres (ex. adresses email, numéros de téléphone).
* 📌 `AUTO_INCREMENT` : Génère automatiquement une valeur numérique séquentielle unique pour chaque nouvel enregistrement inséré.

Ouvre un onglet de requête (*New Query Tab*) dans MySQL Workbench et exécute les blocs d'instructions DDL suivants pour initialiser la base de données de production :

#### Étape 1 : Initialisation de l'espace de données

```sql

-- Activer et utiliser cette base pour les requêtes suivantes

```

#### Étape 2 : Implémentation des tables physiques

```sql
-- 1. Table Customer
CREATE TABLE customer (
    id INT AUTO_INCREMENT PRIMARY KEY,
    firstname VARCHAR(50) NOT NULL,
    lastname VARCHAR(50) NOT NULL,
    email VARCHAR(100) NOT NULL UNIQUE,
    phone VARCHAR(15)
);

-- 2. Table Product
CREATE TABLE product (
    id INT AUTO_INCREMENT PRIMARY KEY,
    product_name VARCHAR(100) NOT NULL,
    description VARCHAR(100) NOT NULL,
    price DECIMAL(10,2) NOT NULL,
    stock INT NOT NULL
);

-- 3. Table Sale
CREATE TABLE sale (
    id INT AUTO_INCREMENT PRIMARY KEY,
    sale_date DATE NOT NULL,
    customer_id INT NOT NULL,
    FOREIGN KEY (customer_id) REFERENCES customer(id)
);

-- 4. Table Sale_Product (Table intermédiaire gérant la relation N:M)
CREATE TABLE sale_product (
    sale_id INT NOT NULL,
    product_id INT NOT NULL,
    quantity INT NOT NULL,
    PRIMARY KEY (sale_id, product_id),
    FOREIGN KEY (sale_id) REFERENCES sale(id),
    FOREIGN KEY (product_id) REFERENCES product(id)
);

```

> [!NOTE]
> **Pourquoi la table `sale_product` est-elle structurée ainsi ?**
> * Elle ne possède pas d'identifiant artificiel propre.
> * Sa clé primaire est dite **composée** car elle réunit les deux clés étrangères.
> * Elle permet de lister un nombre illimité de produits pour une même vente, tout en rattachant la quantité précise vendue pour chaque article.
