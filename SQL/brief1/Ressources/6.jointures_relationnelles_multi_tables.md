### 1. Introduction aux jointures

En SQL, les jointures sont utilisées pour combiner des enregistrements de deux ou plusieurs tables en fonction d'une relation logique (clés primaires et clés étrangères) entre elles.

> [!IMPORTANT]
> Les jointures sont le fondement même des bases de données relationnelles. Elles permettent de décloisonner l'information pour obtenir des analyses métiers complètes.

#### Pourquoi segmenter les données dans des tables distinctes ?
1.  **Structure relationnelle** : Éviter la redondance (répétition inutile des données). Au lieu d'écrire le nom complet d'une catégorie dans chaque ligne de produit, on stocke un simple `category_id`.
2.  **Normalisation** : Organiser logiquement les entités et maintenir l'intégrité des données via des contraintes de clés.
3.  **Besoin d'analyse** : Pour savoir précisément quel client a acheté quel produit et à quelle date, il est indispensable de lier la table des clients, celle des ventes, et celle des produits.

#### Syntaxe générale

```sql
SELECT 
   tableA.colonne1, 
   tableB.colonne2, 
   ...
FROM 
   tableA
JOIN_TYPE 
   tableB 
      ON tableA.colonne_commune = tableB.colonne_commune;

```

* `JOIN_TYPE` définit le comportement d'association (INNER, LEFT, etc.).
* La clause `ON` spécifie la condition de correspondance stricte.

### 2. INNER JOIN : L'intersection

![](https://raw.githubusercontent.com/Salma-AZIZ/Data-Analyst-Juin-2026---Simplon/main/assets/sql/sql_inner_join.png)

`INNER JOIN` (ou jointure interne) retourne uniquement les enregistrements qui possèdent une correspondance exacte dans les deux tables. C'est l'intersection stricte de deux ensembles.

```sql
-- Trouver les informations des clients qui ont effectué au moins un achat
SELECT 
   customer.firstname, 
   customer.lastname, 
   sale.id AS sale_id, 
   sale.sale_date
FROM 
   customer
INNER JOIN 
   sale ON customer.id = sale.customer_id
ORDER BY 
   sale.sale_date DESC;

```

> [!TIP]
> `INNER JOIN` est le type de jointure le plus couramment employé. C'est d'ailleurs le comportement par défaut en SQL : si tu écris le mot-clé `JOIN` sans aucun préfixe, le SGBD l'exécutera comme un `INNER JOIN`.

---

### 3. LEFT JOIN : L'inclusion de la table de gauche

![](https://raw.githubusercontent.com/Salma-AZIZ/Data-Analyst-Juin-2026---Simplon/main/assets/sql/sql_left_join.png)


`LEFT JOIN` (ou `LEFT OUTER JOIN`) retourne **tous** les enregistrements de la table de gauche (Table A). Si aucune correspondance n'est trouvée dans la table de droite (Table B), les colonnes de la Table B contiendront la valeur `NULL`.

```sql
-- Trouver tous les clients, qu'ils aient effectué des achats ou non
SELECT 
   customer.firstname, 
   customer.lastname, 
   sale.id AS sale_id, 
   sale.sale_date
FROM 
   customer
LEFT JOIN 
   sale ON customer.id = sale.customer_id
ORDER BY 
   customer.lastname, customer.firstname;

```

> [!TIP]
> **Technique d'analyse** : Le `LEFT JOIN` est l'outil parfait pour identifier des éléments dits **orphelins** ou inactifs. Pour lister uniquement les clients qui n'ont jamais effectué le moindre achat, il te suffit d'ajouter un filtre restrictif `WHERE sale.id IS NULL` à la requête précédente.

---

### 4. RIGHT JOIN : L'inclusion de la table de droite

![](https://raw.githubusercontent.com/Salma-AZIZ/Data-Analyst-Juin-2026---Simplon/main/assets/sql/sql_right_join.png)


`RIGHT JOIN` fonctionne exactement comme le `LEFT JOIN`, mais la priorité d'inclusion totale est donnée à la table de droite (Table B).

```sql
-- Trouver tous les produits et leurs ventes associées (inclut les produits invendus)
SELECT 
   p.product_name, 
   sp.sale_id, 
   sp.quantity
FROM 
   sale_product AS sp
RIGHT JOIN 
   product AS p ON sp.product_id = p.id
ORDER BY 
   p.product_name;

```

> [!IMPORTANT]
> En production, on privilégie l'utilisation du `LEFT JOIN` pour standardiser l'ordre de lecture logique. La requête ci-dessus s'écrit plus naturellement ainsi :
> ```sql
> SELECT p.product_name, sp.sale_id, sp.quantity
> FROM product AS p
> LEFT JOIN sale_product AS sp ON p.id = sp.product_id
> ORDER BY p.product_name;
> 
> ```
> 
>

### 5. FULL OUTER JOIN : L'union totale

![](https://raw.githubusercontent.com/Salma-AZIZ/Data-Analyst-Juin-2026---Simplon/main/assets/sql/sql_full_outer_join.png)

`FULL OUTER JOIN` retourne l'ensemble des enregistrements dès qu'une correspondance est présente dans l'une des deux tables (combinaison complète du `LEFT` et du `RIGHT`).

> [!WARNING]
> Le SGBD MySQL ne supporte pas nativement la clause `FULL OUTER JOIN`. Pour obtenir ce comportement, on émule l'union totale en combinant un `LEFT JOIN` et un `RIGHT JOIN` à l'aide de l'opérateur de fusion `UNION` :
> ```sql
> SELECT c.firstname, c.lastname, s.id AS sale_id
> FROM customer c
> LEFT JOIN sale s ON c.id = s.customer_id
> UNION
> SELECT c.firstname, c.lastname, s.id AS sale_id
> FROM customer c
> RIGHT JOIN sale s ON c.id = s.customer_id
> WHERE c.id IS NULL;
> 
> ```
> 
>

### 6. CROSS JOIN : Le produit cartésien

![](https://raw.githubusercontent.com/Salma-AZIZ/Data-Analyst-Juin-2026---Simplon/main/assets/sql/sql_cross_join.png)

Le `CROSS JOIN` combine chaque ligne de la première table avec chaque ligne de la deuxième table, sans aucune condition de correspondance.

> [!CAUTION]
> **Attention aux performances** : Si la Table A contient 100 lignes et la Table B contient 50 lignes, le résultat générera instantanément $100 \times 50 = 5000$ lignes. Sur des tables volumineuses, un produit cartésien accidentel peut saturer la mémoire (RAM/CPU) du serveur.

```sql
-- Générer toutes les combinaisons possibles entre les 5 premiers clients et les 3 premiers produits
SELECT 
   c.firstname, c.lastname, p.product_name
FROM 
   (SELECT * FROM customer LIMIT 5) AS c
CROSS JOIN 
   (SELECT * FROM product LIMIT 3) AS p;

```

### 7. Cas pratiques avancés

#### Jointures multiples (Chaînage de tables)

Pour lier plus de deux tables, il te suffit d'enchaîner les clauses `JOIN` les unes après les autres. L'optimiseur du SGBD se chargera de planifier l'ordre d'exécution le plus performant.

```sql
-- Audit d'extraction complet : Qui a acheté quel produit, quand et pour quel montant ?
SELECT 
   c.firstname, c.lastname, s.sale_date,
   p.product_name, p.price, sp.quantity,
   (sp.quantity * p.price) AS total_price
FROM 
   customer c
JOIN sale s ON c.id = s.customer_id
JOIN sale_product sp ON s.id = sp.sale_id
JOIN product p ON sp.product_id = p.id
ORDER BY 
   s.sale_date DESC;

```

#### Auto-jointures (Self-Joins)

Une auto-jointure consiste à joindre une table avec elle-même. Elle est indispensable pour modéliser des structures hiérarchiques ou de dépendance au sein d'une même entité (ex. lister chaque employé avec le nom de son manager, stocké dans la même table).

```sql
SELECT 
   e.name AS employee_name, 
   m.name AS manager_name
FROM 
   employee e
LEFT JOIN 
   employee m ON e.manager_id = m.id;

```
